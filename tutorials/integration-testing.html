Introduction about the weekly game

After reviewing the basics of how to make an API request to OtherScreen in the Getting Started tutorial, you'll want to walk through making some requests to real endpoints to make sure you get how they work.

Rather than having you junk up your own account, or request permissions that you may not need otherwise, we're providing some defined resources on our staging environment that have desirable properties for you to get around to all the places that you'll need to arrive at an implementation of an OtherScreen client.

If you've ever played Portal, these names and references may be familiar to you. We're going to explore interacting with OtherScreen world from the context of a specific small set of users.

Weekly games our 24x7 game

These games will regenerate upon transfer. Our top ten most popular trivia questions will always be used to generate the games on staging.

Hitting the Games Listing endpoint will always show a main game provisioned, which should have ten questions available to answer. If you run through all of them with Chell, you can always Create an Account to receive more auth tokens and continue playing.

Live event the centerpiece

There may be a variety of live events available, but the regenerating one here will be named Portal 3 Launch. This event will have a preroll containing an Announcement, Prop, Poll, and Trivia. That's intended to place all the richness a preroll can offer in a single event. See the Preroll tutorial for more information.

The live event will have a couple of chat messages in it, and a small leaderboard. It should be a good starting place for your live event modelling.

New questions coming soon

The web client has the ability to answer Keyboard Prompts, Keypad prompts, List Prompts, Range Prompts as well as Multiple Choice Prompts.

If you cannot implement answering these question types, you are encouraged to simply pass them over as a part of preroll. At this time, preroll is the only place they can occur.

Groups temporarily shelved

For a variety of reasons, our group implementation never took off the way we'd imagined. We currently have groups as a part of the web client only for administrators, so no group implementation work is currently needed on the part of any client.

That said, if you'd like to implement groups, the Chell user has been invited to two different groups, (one from Cube, and one from Cave). In turn, she can create and invite Cube and Cave to groups as needed.
