Getting Started

How to make requests to the OtherScreen API

HTTP Requests details great and small

OtherScreen intends to build a social TV platform with an API easily understood and utilized from every language and environment.

We are starting with web and iOS client implementations, but we'll be using examples from curl throughout this documentation.

The following tips are to help get you started, and we'll explain them one by one.

  • HTTPS only
  • Domain name
  • API Versioning
  • Client Identification
  • Authentication

Quick Start

All requests should be HTTPS and made to https://app.otherscreen.com/

All requests must include an accept header application/vnd.otherscreen.com; version=1

All requests must include a unique user-agent header that indicates client platform, and version number for the client implementation.

An example User-Agent header would be OtherScreen iOS 1.3.1

To authenticate a user, acquire an authorization token and set the HTTP_X_OTHERSCREEN_AUTH_TOKEN header to that token value.

HTTPS Secure transmission by default

Since the API allows account sign-in and account creation with a password, we choose to secure our entire transmission with clients.

Even a single request made over HTTP can allow a man-in-the-middle to hijack session information and gain access to a user profile.

While requests made via HTTP are not yet rejected, they may be without elevating the API version number. We will not consider this a breaking change.

API Versioning our API guarantees

Our 1.0 API indicates a commitment by our technical team to never compromise the behavior of your client implementation for the lifetime of that API version. At this time, we have no plans to ever retire an API.

To take advantage of this decision, you must indicate which version of the API that you intend to use with an accept header. If, for any reason, our engineers introduce new or amend existing data forms, we pledge to increment the API version required in this accept header to receive the new data form.

In other words, when you use an accept header that indicates you want version 1.0 of the API, we'll never introduce a change in the form of the result you get back.

At this time, there are no other choices for API version, so make your Accept header application/vnd.otherscreen.com; version=1

Client Identification we're not ready for API tokens

Enterprise level API's such as Twitter can require an API token for use. This allows Twitter to defend itself against malicious or incidental bad behaviors by clients. Our API does not yet meet that level of need, but we intend to.

What is vital for us to understand, however, is for us to understand which clients are currently being used by our players-- as well as which are under development and which features are being used by which implementation.

To that end, you are asked to self-select a unique string that describes the client you are building. Platform, version, and author may all be related details. At this time, the only prohibited choice is com.otherscreen.OtherScreen/5 which has been taken.

Authentication making sure we know who's playing

Much of the OtherScreen API requires a specific user to be signed in to return proper responses. There are some choices regarding authentication, all of which result in receiving an authentication token that uniquely identifieds a user.

Use that authentication token in an OtherScreen-unique request header HTTP_X_OTHERSCREEN_AUTH_TOKEN which has been taken.
