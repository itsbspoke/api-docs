Listing what games can I currently play?

Reason for existence

It returns a listing of live events that the client will have an opportunity to play. After authenticating, this should be the first request that clients make.

Planned We plan to make a version of this endpoint that does not require authentication in the future.

Check-In basic event orientation and information

Reason for existence

A user must make this call to begin participating in an event, and then periodically as they continue participation in the event in question.

This call returns at least two and possibly three entities with useful information:

  • The Event Card-- this is used when calculating and displaying a user's score and statistics for a particular event
  • The leaderboard-- this is an array of event cards ordered by their current competitive rank. In this context, these event cards also contain information about the recency of their owner's participation, so that each user can be fully aware of who is currently engaged in gameplay and who is not.
  • Preroll-- in some cases, there may be some content available to display to the user, usually but not always before the start time of the event in question. See Completing preroll for more information.

Send a chat

Reason for existence

Humans like talking to each other. This helps complete the human experience of playing our game.

Invocation form

POST /live_events/[id]/messages

The [id] parameter should be one available from the listing call detailed above.

body
The body of the chat. Limited to 255 characters

Status

Complete

HTTP Error Codes

400
Make sure you are passing in a message body less than 256 characters
401
You must pass an authentication token.
404
Make sure you are pulling the live event id from the listing API call.

Tweet a chat

Reason for existence

Allows a user to share OtherScreen content with their friends and followers on Twitter.

Invocation form

POST /live_events/[live_event_id]/messages/[id]/tweet

This URL should be available in the chat

The [live_event_id] parameter should match the one from the live event listing.

The [id] parameter should be one available from the chat itself, from the history or a WebSocket event.

Status

Planned
